# ADR-004: 감시단 7종 구조 (G1~G7)

- 날짜: 2026-03-16 (G1~G3 v0.1), 2026-03-17 (G2 2인 체제 v0.2), 이후 G4~G7 추가
- 상태: 채택됨
- 결정: 코드/문서 작업 후 7종 감시단이 자동으로 품질 검사를 실행한다.
  - G1 기술: 보안, 코드 품질, Build vs Buy (하위직원 4명)
  - G2 철학: 비전 왜곡 방지. G2-A(비전) + G2-B(존재론) 2단계 검증 (하위직원 4명)
  - G3 규칙: L1/L2 위반, 데이터 규칙, 로그 형식 (하위직원 2명)
  - G4 글쓰기: 대외 산출물 글 품질, 비트겐슈타인 원칙 (하위직원 2명)
  - G5 경영: BM, 재무, 시장 검증, G2 견제 (하위직원 3명)
  - G6 마케팅: 5초 테스트, 욕구 자극, G2 견제 (하위직원 2명)
  - G7 디자인: 철학 시각화, 모션 품격 (하위직원 2명)
- 이유:
  - "전략 판단 대부분을 Claude한테 위임. 검증 구조 없음"이 T9 OS 구축 동기 중 하나였다. 감시단은 AI 자기 검증 체계이다.
  - project-alpha 프로젝트에서 3종 감시단(기술/철학/규칙)을 먼저 실험한 뒤 T9OS 전체로 확장했다.
  - G2를 2인 체제(비전+존재론)로 분리한 이유: G2-B에서 "dissolved=폐기"가 시몽동 위반임을 발견한 것이 계기였다. 설계자이 가공한 프로젝트 비전(G2-A)과 순수 시몽동 원칙(G2-B) 모두 통과해야 한다.
  - G5/G6이 G2를 견제하는 구조: 철학 감시단이 모든 마케팅을 차단하면 사업이 죽는다. Guardian Debate로 충돌 해소.
  - 프로젝트별 ANCHOR 문서로 필수어/금지어 기반 기계적 검증이 가능하다.
- 대안:
  - **코드 리뷰만 (G1만)**: 비전 왜곡, 규칙 위반을 잡지 못함 — 불충분.
  - **인간 리뷰**: 설계자 혼자로는 모든 산출물 검토 불가 — 보완책으로만 사용.
  - **단일 종합 감시단**: 역할이 뭉뚱그려지면 전문성 저하 — 폐기.
- 결과:
  - `constitution/GUARDIANS.md`에 전체 규칙이 문서화되어 있다.
  - 경량(G1만)/기본(G1+G2+G3)/전체(G1~G7, 21명) 3단계 모드로 운영.
  - P0/CATASTROPHIC 발견 시 즉시 수정 (설계자 승인 불요).
  - 로그/.claude 메타파일/git 작업은 감시단 생략 대상.
  - 감시단 간 토론(Guardian Debate): 합의 불가 시 설계자이 최종 판단.

## Simondon Mapping
이 결정이 시몽동의 어떤 원리를 구현하는가: 자기-조절(auto-regulation) — 체계가 스스로의 산출물을 검증하여 준안정 상태를 유지하는 되먹임 구조.
