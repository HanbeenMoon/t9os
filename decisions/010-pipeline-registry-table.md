# ADR-010: 파이프라인 레지스트리 테이블 방식

- 날짜: 2026-03-16
- 상태: 채택됨
- 결정: 모든 파이프라인/도구를 마크다운 테이블 형태로 CLAUDE.md, L1, BIBLE.md, memory에 동시 등록한다. 새 파이프라인 생성 시 반드시 3곳 이상을 동시 갱신한다. "기록 안 하면 = 안 만든 것."
- 이유:
  - 멀티 세션 운영에서 가장 큰 문제: 한 세션이 만든 도구를 다른 세션이 모른다. 파이프라인이 존재하지만 발견되지 않으면 중복 Build가 발생한다.
  - 재현성 규칙: "다른 세션이 발견 못하면 존재하지 않는다." 이것은 L1/L2 위반 불가 규칙이다.
  - 파이프라인 레지스트리 DB를 별도로 Build하는 것은 SRBB 원칙 위반이다. 마크다운 테이블 + 사람이 읽을 수 있는 형태가 가장 단순하고 효과적이다.
  - 현재 14개 파이프라인(`pipes/`), 4개 라이브러리(`lib/`), 14개 스킬(`.claude/skills/`)이 등록되어 있다.
- 대안:
  - **별도 레지스트리 DB**: Build 오버헤드, 조회 도구도 만들어야 함 — SRBB 위반.
  - **코드 주석만**: 세션 시작 시 코드를 전부 읽어야 함, 토큰 낭비 — 폐기.
  - **자동 발견 (introspection)**: 구현 복잡도 높음, 동작 설명은 자동 추출 불가 — 불채택.
  - **단일 위치만 기록**: 위치를 모르면 발견 불가, 3곳 동시 기록이 중복 같지만 접근 경로가 다르므로 필요 — 폐기.
- 결과:
  - CLAUDE.md 섹션 10에 파이프라인 테이블, BIBLE.md 섹션 6.4에 상세 테이블.
  - L1에 "파이프라인 레지스트리 (재현성 필수)" 섹션.
  - memory에 `tool_t9_pipes_overview.md`, `tool_gm_batch_guardian.md` 등 도구별 가이드.
  - `pipes/reproducibility_check.py`가 재현성 자동 체크를 실행한다.
  - G3 규칙 감시단이 새 도구의 레지스트리 등록 여부를 감사한다.

## Simondon Mapping
이 결정이 시몽동의 어떤 원리를 구현하는가: 전도(transduction) — 한 영역의 기록이 다른 세션/영역으로 전파되어 발견 가능해지는 정보 전달 구조.
