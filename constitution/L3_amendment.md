## 개정 트리거

1. **현실 불일치**: 규칙이 현실 작업과 맞지 않는 사례 3회 이상

4. **성능 저하**: 토큰 낭비, 산출물 품질 하락, 메타 작업 비율 증가
5. **구체화 필요**: 도구 간 통합 수준이 변화했을 때 (아래 참조)

## 구체화 수준 측정 (Concretization)

시몽동의 "구체화": 추상적(각 부품 독립) → 구체적(부품 간 다기능적 통합)으로 진화.
T9 OS의 구체화 수준을 주기적으로 평가:

| 수준 | 설명 | 예시 |
|------|------|------|
| 1-추상 | 각 도구가 독립 실행, 수동 전달 | cc가 결과를 복붙해서 cx에 전달 |
| 2-연결 | 도구 간 자동 연결 존재 | cc가 cx를 직접 호출, 로그 자동 저장 |
| 3-통합 | 하나의 작업이 여러 도구를 자동 횡단 | 세션 종료 시 로그+memory+노션 동시 갱신 |
| 4-구체 | 도구 경계가 사라짐, 하나의 유기체처럼 작동 | 환경 변화 → 자동 재라우팅 → 산출물 |

현재 수준: **2-연결** (cc↔cx 호출, 로그 자동화는 되나 환경 감지는 수동)
목표: **3-통합** (이번 분기)

## 변조 원칙 (Modulation)

- 개정은 "오류 수정"이 아니라 **연속적 변조**의 일부
- 규칙이 경직되었다는 느낌 자체가 개정 트리거
- 형식적 완결성을 추구하지 않음 — 가변성이 목표
- 개정 빈도가 너무 높으면(주 3회+) → 상위 구조 문제, L3 자체를 점검

## 비소급 원칙 (Non-retroactivity / Rückwirkungsverbot)

<!-- 법이론 근거: Hans Kelsen, Pure Theory of Law (1934/1960)

     핵심 가치: 법적 안정성(Rechtssicherheit) — 행위자가 행위 시점의 규칙을

### 원칙

L1/L2/L3 규칙이 개정되더라도, **개정 이전에 수행된 행위는 개정 시점의 규칙이 아니라 행위 시점의 규칙으로 평가한다.**

이것은 단순히 "과거를 무효화하지 않는다"가 아니다. 핵심은 **계획 가능성(planability)**이다. cc가 세션 중에 L1 규칙을 믿고 100개의 결정을 내렸는데, 세션 종료 후 규칙이 바뀌어서 그 100개가 전부 위반이 된다면 — cc는 어떤 행동도 확신을 갖고 할 수 없게 된다. 이것은 시스템을 마비시킨다.

### 적용 규칙

1. **세션 내 안전 보장**: L1/L2가 세션 도중에 변경되더라도(다른 세션에서 개정, live-digest를 통해 전파), 현재 세션은 **세션 시작 시점의 규칙 버전**으로 남은 작업을 완료할 수 있다. 단, 새 규칙을 인지한 시점 이후의 **새로운 작업**에는 새 규칙을 적용한다.

2. **감시단 소급 금지**: 규칙 개정 후 감시단(G3)이 과거 커밋/산출물을 재검사할 때, **행위 시점에 존재하지 않았던 규칙**으로 위반을 판정하지 않는다. 개정 이전 커밋의 감사 기준은 해당 커밋 시점의 schema_version이다.

3. **예외 — 소급 적용이 허용되는 경우**:
   - 보안 취약점 (OWASP P0): 보안은 규칙 이전에 존재하는 물리적 위험이므로 소급 수정 가능
   - 한빈의 명시적 소급 지시: "과거 것도 전부 고쳐" 같은 직접 명령
   - 존재론 위반 (G2-B VIOLATION): 시몽동 원칙은 L1보다 상위이므로 소급 적용

4. **개정 로그에 소급 여부 명시**: 개정 로그 작성 시 `소급: YES/NO` 필드를 추가한다. 기본값은 NO.

### 위반 시

소급 적용으로 인한 위반 판정이 발생하면, cc는 해당 판정을 **유보(suspend)**하고 한빈에게 보고한다. 자동 수정하지 않는다.
