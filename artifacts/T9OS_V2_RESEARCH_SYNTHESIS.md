| 문제 | 현황 | 영향 |

### 1-3. 한빈의 구조적 동형성으로 해석

> "시장이 너무 성공하면 연결된 다른 시장이 실패. 과적합. 적당히 실패해야 한다."

- **v1 = 과도한 규제를 받는 시장**: 규칙 54개 = λ가 너무 큼 = 과소적합 = 시스템 경직
- **AT1 30분 완풀 = λ≈0**: 규칙 최소화 + AI에 위임 = 최적 정규화
- **v2의 목표**: 최적 λ를 찾는 것. 규칙을 줄이되, 외부 정책 엔진으로 핵심만 강제.

## 2. 설계 원칙 (리서치에서 도출)

### 원칙 1: 역할 분리 (Generator-Critic)
- cc가 혼자 전부 하지 않는다
- 실행하는 cc ≠ 검증하는 계층
- 기업 동형성: CEO ≠ 감사팀 ≠ 실무팀

### 원칙 2: Pre-Action Authorization (2층 — 죽은 것 + 살아있는 것)
- LLM 행동을 실행 전에 외부에서 가로채서 검증
- **하드 게이트**: `type: command` — 물리 법칙처럼 절대적 (정적, 바뀔 필요 없음)
- **소프트 게이트**: `type: prompt/agent` — 별도 LLM이 맥락 판단 (동적, 사례 축적으로 진화)
- 정책 엔진 전체가 "살아있는 시스템"이어야 함 (한빈 피드백 2026-03-20)

### 원칙 3: 최적 정규화 (λ 최적화)
- 규칙은 5개 이하로 줄이고, 나머지는 코드로 강제
- "인간의 판단이 들어가야 한다는 것 자체가 사고의 한계" (한빈)
- 한빈 개입 = 정규화항. 과하면 과소적합, 없으면 과적합

### 원칙 4: 설계자 모드
- "나는 주인공이 아니고 설계자다" (한빈 2026.02.03)
- 한빈은 방향과 구조를 설계. cc는 배경에서 실행
- v1의 "사용자-도구" 관계 → v2의 "설계자-시스템" 관계

### 원칙 5: 트로이목마
- "기술은 드러나고 철학은 느껴지는 것" (한빈)
- 백엔드 복잡함 ↔ 프론트엔드 단순함
- 한빈에게 보이는 것: 대시보드 7개 KPI + 다이어그램
- 숨겨진 것: 시몽동, VSM, OODA, 정책 엔진

### 원칙 6: Strangler Fig (마이그레이션)
- 전면 재작성 금지. 잘 되는 건 놔두고 문제만 교체
- 한 번에 한 모듈만. 매 단계마다 검증
- v1과 v2 공존 기간 최소 2주

### 원칙 7: 점진적 공개 (Progressive Disclosure)
- 한빈의 인지 용량 존중하되, 인지에만 맞추면 병목
- Level 0: 비유/메타포 → Level 1: 시각적 맵 → Level 2: 의사결정 참여 → Level 3: 패턴 습득
- C4 Model의 줌 인/아웃 패턴

## 3. 아키텍처 방향 (리서치 기반)

### 3-1. v1 → v2 핵심 전환

```
v1: 한빈 → cc(전부) → 결과
v2: 한빈(설계자) → 정책엔진(강제) → cc(실행) → 검증계층 → 결과
```

### 3-2. 4개 계층

| 계층 | 역할 | 구현 방향 |
|------|------|-----------|
| **한빈 계층** | 방향, 비전, 최종 승인 | 대시보드 + 텔레그램 + RFC |
| **정책 계층** | L1 핵심 규칙 물리적 강제 | PreToolUse 훅 + deterministic 코드 |
| **실행 계층** | 코드, 문서, 분석, 오케스트레이션 | cc + cx + gm (현행 유지) |
| **검증 계층** | 사전/사후 검증 | 감시단 사전 전환 + Agent Teams critic |

### 3-3. 기존 자산 활용 (Build vs Buy)

| 컴포넌트 | 현황 | v2 전략 |
|-----------|------|---------|
| t9_seed.py | 잘 작동 | **유지** + 확장 |
| .t9.db (SQLite) | 안정적 | **유지** + 스키마 확장 |
| 감시단 (gm_batch.py) | 사후 리뷰 | **전환**: 사전 검증으로 |
| 파이프라인 16개 | 돌아감 | **그룹화**: 라우터 패턴 |
| IPC (ipc.py) | 기본 동작 | **강화**: JSON inbox 패턴 |
| 세션 훅 | 시작/종료만 | **확장**: PreToolUse 추가 |
| 헌법 (L1/L2/L3) | 과다 | **축소**: 핵심 5개 + 코드 강제 |
| memory 54개 | 과다 | **정리**: 핵심 유지, 나머지 통합 |

## 4. 구조적 동형성 매핑 (완료)

6개 분야에서 T9 OS v2 4계층에 매핑되는 패턴 추출 완료.

| 분야 | 핵심 패턴 | T9 OS 설계 시사점 |
|------|-----------|------------------|
| **정치학** | 삼권분립, 견제와 균형 | 감시단을 cc와 별도 프로세스로 분리 (사법 독립) |
| **군사학** | OODA(Orient 핵심), Auftragstaktik(임무형 지휘) | Orient 연속화 + Commander's Intent 3줄 표준 |
| **생물학** | 자율신경계(교감/부교감), 면역계(자기/비자기) | 가속+제동 항시 작동, 감시단=면역계 |
| **경영학** | VSM(5개 시스템), 도요타(안돈/지도카) | **S4(전략탐색) 구축 최우선**, P0 자동정지 |
| **생태학** | 스티그머지(간접통신), 벌떼(정족수) | 환경 기반 통신 강화, 가변 정족수 |
| **시스템이론** | 오토포이에시스, 사이버네틱스 | T9 OS는 이미 자기생산적, 의식적 강화 필요 |

**가장 높은 동형성: VSM** — Beer의 S1~S5가 T9 OS 4계층에 직접 매핑. S4(외부 환경 자동 탐색)가 가장 약한 고리.

상세: `T9OS_V2_PLAN.md` 섹션 8 참조.
- 철학(시몽동, 오토포이에시스) → 자기 생산 시스템
