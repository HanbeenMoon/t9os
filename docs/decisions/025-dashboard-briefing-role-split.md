# ADR-025: Dashboard/Briefing 역할 분리

- 날짜: 2026-03-17
- 상태: 채택됨
- 결정: T9 Dashboard를 두 페이지로 분리한다. Dashboard(`/`)는 상세 운영 정보(마감일 14일, 캘린더 전체, 인프라, 최근 전이 10개), Briefing(`/briefing`)은 핵심 요약(D-3 긴급, 오늘 일정, 긴급 5개)을 표시한다. 각 섹션에 (?) 툴팁을 추가하여 비개발자도 의미를 이해할 수 있게 한다.
- 이유:
  - 초기 Dashboard는 모든 정보를 한 페이지에 표시하여 정보 과부하가 발생했다.
  - "숲 먼저, 나무 나중" 원칙에 따라 Briefing은 10초 안에 현재 상황을 파악할 수 있어야 한다.
  - Dashboard는 상세 분석이 필요할 때만 사용한다. 평소에는 Briefing만 보면 충분하다.
  - 툴팁 추가로 각 섹션의 의미("이 숫자가 왜 중요한가")를 즉시 파악 가능.
- 대안:
  - **단일 페이지 유지**: 정보 과부하, 핵심 파악 어려움 — 폐기.
  - **3개 이상 페이지**: 관리 복잡도 증가, 2개로 충분히 커버 — 불채택.
  - **사용자 커스터마이징**: 구현 복잡도 대비 효용 낮음 — 불채택.
- 결과:
  - Briefing은 모바일에서도 핵심만 확인 가능한 경량 뷰.
  - Dashboard는 데스크톱에서 상세 분석용.
  - Tooltip.tsx 컴포넌트를 CSS hover로 구현 (외부 라이브러리 없음, SRBB 원칙).
  - Nav 컴포넌트에 Dashboard/Briefing/Architecture/Integrity 네비게이션.

## Simondon Mapping
이 결정이 시몽동의 어떤 원리를 구현하는가: 정보의 계층적 개체화 — 동일한 데이터가 두 가지 수준(요약/상세)으로 개체화되어, 사용자의 현재 필요(빠른 파악 vs 깊은 분석)에 맞는 형태로 제공된다.
